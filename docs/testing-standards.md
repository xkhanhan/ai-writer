# 测试规范

> **适用于：代码审计员角色（必读）。其他角色可选阅读。**
> **核心原则：禁止只测正向流程。每个任务必须覆盖以下全部 7 类场景。**

---

## 一、测试场景类型（必须全部覆盖）

每个任务的测试用例，必须按以下 **7 类场景** 分别编写用例。缺少任何一类视为测试不完整。

### 1. 正向流程（Happy Path）

> 用户按预期方式操作，一切正常。

**必测项：**
- 用户完成核心功能的完整路径
- 所有交互元素（按钮、输入框、下拉框）按预期工作
- 数据正确展示、正确保存、正确返回

**示例：**
- 点击保存 → 数据持久化 → 刷新后仍在
- 选择选项 → 界面响应 → 结果正确

### 2. 反向流程（Negative / Error Path）

> 用户做了错误操作、异常操作、或系统出错时的处理。

**必测项：**
- 空值/空列表时的展示（0 条数据、未选择任何项）
- 必填字段为空时的提示
- API 返回错误码时的前端表现（400/401/403/500）
- 网络断开/超时时的降级处理
- 删除已删除的资源、编辑不存在的记录

**示例：**
- 不填必填项点保存 → 应有错误提示，不应静默失败
- API 返回 500 → 应显示友好错误信息，不应白屏
- 列表为空时 → 应显示空状态提示

### 3. 边界条件（Boundary / Edge Cases）

> 极端值、极长/极短输入、临界状态。

**必测项：**
- 输入 0 个字符、1 个字符、最大长度字符
- 数字输入：0、负数、极大值、小数
- 列表只有 1 项时的操作
- 列表有 100+ 项时的滚动/性能
- 时间相关：跨天、跨月、闰年
- 唯一性：重复名称/ID 的处理

**示例：**
- 输入框填 1000 字符 → 不应溢出/截断异常
- 列表只有 1 项时删除 → 不应报错
- 创建同名记录 → 应有冲突提示或自动编号

### 4. 状态流转（State Transitions）

> 功能涉及多个状态时，每条状态转换路径都必须验证。

**必测项：**
- 所有合法状态转换（正常流转）
- 非法状态转换的拒绝（如从"已完成"再编辑）
- 状态回退/撤销
- 并发操作的状态冲突

**示例：**
- 任务从"待开发"→"开发中"→"待测试"→"测试结束" 全链路
- 从"测试结束"试图再次编辑 → 应拒绝
- 两人同时编辑同一资源 → 应有冲突处理

### 5. 交互细节（Interaction Details）

> UI 组件的具体交互行为，容易被忽略但影响用户体验。

**必测项：**
- 键盘操作：Tab 切换、Enter 提交、Escape 关闭
- 快速连续点击（防重复提交）
- 加载状态（loading spinner/skeleton）
- 禁用状态（按钮不可点击时的视觉+行为）
- 悬停效果（hover 态）
- 焦点状态（focus 态）
- 拖拽操作的边界（拖到无效区域）
- 滚动行为（长列表滚动、固定头部/底部）

**示例：**
- 快速双击保存按钮 → 不应创建两条记录
- 数据加载中点击按钮 → 按钮应显示 loading 且不可重复点击
- Tab 键遍历所有交互元素 → 焦点顺序合理

### 6. 数据一致性（Data Consistency）

> 前后端数据的完整性、持久化、同步。

**必测项：**
- 操作后刷新页面，数据不丢失
- 操作后切到其他页面再回来，数据正确
- 多窗口/多 Tab 下同一数据的同步
- 修改 → 撤销 → 再修改，数据正确
- 关联数据的级联更新（如删除卷时其下章节的处理）

**示例：**
- 创建记录 → 刷新 → 记录仍在
- 编辑 A 页面数据 → 切到 B 页面 → 回到 A 页面 → 修改仍在
- 删除父记录 → 子记录的处理（级联删除/保留/解绑）

### 7. 跨模块影响（Cross-module Impact）

> 改动是否影响其他模块，依赖链是否断裂。

**必测项：**
- 本次改动涉及的共享组件在其他页面的表现
- 修改 API 后，所有调用方的兼容性
- CSS 变量/主题修改后，所有引用处的表现
- 类型定义修改后，下游消费者的兼容性

**示例：**
- 修改了 `SplitPanel` 组件 → 所有使用 SplitPanel 的页面都要验证
- 新增了 CSS 变量 → 检查旧变量消费者是否受影响
- 修改了 API 响应格式 → 所有前端消费方是否兼容

## 二、测试用例编写规范

### 格式要求

测试用例使用 Markdown 表格格式，文件存放于 `docs/test-cases/` 目录，按任务 ID 命名：

```
docs/test-cases/F-010-variable-validation.md
docs/test-cases/CSS-003-important-cleanup.md
```

每个测试用例必须包含以下字段：

| 字段 | 必填 | 说明 |
|------|------|------|
| **PRD 来源** | ✅ | 必须标明此用例对应的 PRD 文件 + 章节/表格行/验收标准编号。无 PRD 依据的用例禁止编写 |
| **用例编号** | ✅ | 格式：`TC-{任务ID}-{场景前缀}-{序号}`，如 `TC-F010-H01` |
| **用例标题** | ✅ | 一句话描述测试目的 |
| **前置条件** | ✅ | 执行测试前必须满足的状态 |
| **操作步骤** | ✅ | 编号列出每一步操作，精确到按钮名称/URL/输入值 |
| **预期结果** | ✅ | 每步操作对应的期望行为，引用 PRD 具体规定 |
| **测试类型** | ✅ | 功能测试 / 接口测试 / 视觉测试 / 兼容性测试 / 性能测试 |
| **优先级** | ✅ | P0（阻断）/ P1（严重）/ P2（一般）/ P3（建议） |
| **实际结果** | 执行后填写 | 实际观察到的行为，附截图路径/DOM 快照 |
| **通过/失败** | 执行后填写 | ✅ 通过 / ❌ 失败 / ⚠️ 部分通过 |

### PRD 来源必填规则

> **禁止编写无 PRD 依据的测试用例。**

- 每条用例必须能追溯到 PRD 中的具体需求点
- PRD 来源格式：`PRD文件: 章节编号/表格行/验收标准编号`
- 示例：
  - ✅ `PRD: prompt-library-redesign.md §2.3 变量展示规则 — "点击变量即复制 ${variableName} 到剪贴板"`
  - ✅ `PRD: prompt-library-redesign.md AC-5 — "选书→预览Tab→完整提示词渲染正确，变量已替换"`
  - ❌ `PRD: 无`（禁止）

### 反面示例（禁止这样写）

- ❌ 用例没有 PRD 来源 — 无法判断是否应该存在
- ❌ 预期结果写"正常" — 太模糊，什么叫正常？
- ❌ 用例凭直觉编写 — 如"复制预览功能"但 PRD 中无此需求
- ❌ 编辑回显场景遗漏 — 编辑器加载数据是基础功能，必须有对应用例

### 用例编号规则

```
TC-{任务ID}-{场景前缀}-{序号}

场景前缀：
  H = Happy Path（正向）
  N = Negative（反向）
  B = Boundary（边界）
  S = State（状态流转）
  I = Interaction（交互细节）
  D = Data（数据一致性）
  X = Cross-module（跨模块）
```

**示例：**
- `TC-F010-H01` — F-010 正向流程第 1 条
- `TC-F010-N01` — F-010 反向流程第 1 条
- `TC-F010-B01` — F-010 边界条件第 1 条

### 用例示例

```markdown
### TC-F010-N01: 不填必填项保存时的提示

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 进入提示词库 | 页面加载正常 |
| 2 | 选择一个系统默认模板 | 右侧编辑器显示模板内容 |
| 3 | 点击"复制为自定义" | 创建自定义副本，进入编辑状态 |
| 4 | 清空编辑器所有内容（0字符） | 编辑器为空 |
| 5 | 点击"保存"按钮 | 应有错误提示"模板内容不能为空"，不应保存成功 |

### TC-F010-B01: 超长模板内容保存

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 进入提示词库，复制为自定义 | 进入编辑状态 |
| 2 | 粘贴 50000 字符的模板内容 | 编辑器接受输入 |
| 3 | 点击"保存" | 保存成功或提示长度限制，不应崩溃 |
| 4 | 刷新页面 | 内容完整保留（或截断提示） |
```

## 三、测试覆盖维度（必查清单）

每个任务测试时，必须从以下 **10 个维度** 逐一验证，缺一不可：

| 维度 | 检查内容 | 验证方法 |
|------|---------|---------|
| **1. 功能正确性** | 核心功能是否按需求文档实现 | 手动操作 / Chrome DevTools |
| **2. 接口一致性** | 前后端 API 契约是否匹配 | Network 面板 / grep 检查 |
| **3. 状态管理** | 状态流转是否符合预期 | 操作后刷新页面验证持久化 |
| **4. 边界条件** | 空值/异常值/极端输入的处理 | 输入空字符串、特殊字符、超长文本 |
| **5. 主题兼容** | 4 套主题下视觉是否正常 | 切换 warm-paper / cool-gray / clean-white / dark |
| **6. 响应式布局** | 375px / 768px / 1024px 断点表现 | Chrome DevTools 设备模拟 |
| **7. 无障碍性** | aria-label / role / tabIndex 是否完备 | a11y tree 检查 |
| **8. CSS 规范** | 无 `!important` / `:global(.ant-xxx)` / 硬编码颜色 | grep 代码扫描 |
| **9. 数据一致性** | 修改后刷新页面数据不丢失 | 操作 → 刷新 → 对比 |
| **10. 连锁影响** | 改动是否影响其他模块 | 检查依赖链中的组件 |

## 四、测试执行流程

### Phase 1: 静态检查（代码层面）

```bash
# 1. 类型检查
npm run typecheck

# 2. Lint 检查
npm run lint

# 3. 构建检查
npm run build

# 4. CSS 规范扫描
grep -rn "!important" app/ shared/ --include="*.css" | grep -v globals.css
grep -rn ":global(.ant-" app/ shared/ --include="*.css"
grep -rn "rgba(" app/ shared/ --include="*.css" | grep -v globals.css
grep -rn "#ff4d4f" app/ shared/ --include="*.css"
grep -rn "var(--bg-base" app/ shared/ --include="*.css"

# 5. 内联样式扫描
grep -c "style={" app/pages/books/components/creation-zone/components/*/index.tsx

# 6. 死代码/未使用变量扫描
grep -rn "TODO\|FIXME\|HACK\|XXX" app/ shared/ server/ --include="*.ts" --include="*.tsx"
```

### Phase 2: 接口测试（API 层面）

```bash
# 1. 验证 API 返回格式
curl -s http://localhost:3001/api/ai/templates | jq .
curl -s http://localhost:3001/api/books | jq .

# 2. 验证错误处理
curl -s -X POST http://localhost:3001/api/ai/templates -H 'Content-Type: application/json' -d '{}' | jq .
```

或使用 Chrome DevTools MCP 检查：
- 请求是否发出
- 状态码是否正确（200/400/500）
- 响应体结构是否符合前端预期
- 是否有未处理的 404/500

### Phase 3: 功能测试（浏览器层面）

使用 Chrome DevTools MCP (`mcp__chrome-devtools__*`) 执行：

1. **导航到目标页面**
2. **执行测试用例中的操作步骤**
3. **每步操作后截图存档** (`mcp__chrome-devtools__take_screenshot`)
4. **检查控制台错误** (`mcp__chrome-devtools__list_console_messages`)
5. **检查网络请求** (`mcp__chrome-devtools__list_network_requests`)
6. **验证 DOM 状态** (`mcp__chrome-devtools__take_snapshot`)

### Phase 4: 主题/兼容性测试

切换 4 套主题逐一验证：
1. 点击右上角主题切换按钮
2. 每套主题下截图
3. 检查颜色/对比度/阴影是否正常

## 五、BUG 记录规范

> **原则：BUG 描述必须让开发者无需追问即可定位和修复。**

### BUG 标题要求

- 格式：`{模块} {具体现象}`
- ✅ 好：`提示词库 复制为自定义后原始系统默认模板从列表消失`
- ❌ 差：`复制有问题`、`列表不对`、`功能异常`

### BUG 正文必填字段

每个 BUG 必须包含以下 **8 个字段**，缺一不可：

```markdown
🔴 BUG-{序号}: {标题}（{严重度}）

【需求依据】
明确指出违反了 PRD 哪一条。格式：PRD 文件名 + 具体章节/验收标准编号
例：PRD docs/prompt-library-redesign.md §2.6 规定"原始模板应保留在列表中"

【复现步骤】
编号列出每一步操作，精确到点击哪个按钮、选择哪个选项
例：
1. 进入提示词库（http://localhost:3001/?view=settings）
2. 点击左侧"提示词库"
3. 点击"正文生成 global"（确认是系统默认模板）
4. 点击工具栏"复制为自定义"按钮
5. 观察左侧函数列表

【预期行为】
按照 PRD 应该发生什么
例：左侧列表应同时显示"正文生成 global"和"正文生成 edit"两个条目

【实际行为】
实际发生了什么，附上截图路径或 DOM 快照中的关键信息
例：左侧只显示"正文生成 edit"，"正文生成 global"消失。截图：docs/test-cases/screenshots/B-025-list.png

【涉及文件】
列出最相关的代码文件和行号（如果能定位的话）
例：
- app/pages/books/components/prompt-library/index.tsx:200 — getTemplates 函数
- app/pages/books/components/prompt-library/prompt-list.tsx:45 — 列表渲染逻辑

【根因分析】
代码层面的原因（如果能分析出来）
例：getTemplates API 返回的模板列表中，同一个 functionKey 只返回一个模板（优先 custom），没有同时返回 global 和 custom

【影响范围】
这个 BUG 影响了哪些功能/页面/用户场景
例：用户无法切换回系统默认模板查看原始内容；如果误编辑自定义模板，无法对比原始版本

【修复建议】
具体的修复方向（不写具体代码，只写思路）
例：getTemplates 应返回同一 functionKey 下所有模板（global + custom），列表组件按 functionKey 分组渲染
```

### 严重度定义

| 严重度 | 定义 | 示例 |
|--------|------|------|
| 🔴 CRITICAL | 功能完全不可用，阻断核心流程，无 workaround | 预览替换失效、API 返回 500、页面白屏 |
| 🟠 HIGH | 功能异常，有 workaround 或影响多处 | 复制后原模板消失、保存无校验 |
| 🟡 MEDIUM | 功能不符合预期但不阻断主流程 | 响应式布局异常、hover 效果缺失 |
| 🟢 LOW | 体验问题，不影响功能 | aria-label 缺失、CSS 死代码 |

### 反面示例（禁止这样写）

```
❌ "变量面板位置不对" — 开发不知道哪里不对，应该写什么位置
❌ "预览有问题" — 太模糊，哪个功能的预览？什么问题？
❌ "列表显示异常" — 什么异常？多显示了？少显示了？格式错了？
❌ "功能缺失" — 哪个功能？在 PRD 哪里定义的？
```

## 八、测试准入/准出标准

**准入条件（开始测试前必须满足）：**
1. 任务状态为 **待测试**
2. 开发工程师已完成 `typecheck + lint + build`（三项全通过）
3. 代码已合并到 master 分支

**准出条件（任务可标记为测试结束）：**
1. 所有 P0/P1 用例通过
2. 无 CRITICAL/HIGH 级别未修复 BUG
3. 10 个覆盖维度全部验证完毕
4. 7 类测试场景全部覆盖
5. 4 套主题下视觉无异常

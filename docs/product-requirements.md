# 产品需求稿 — AI 辅助小说创作系统

> 最后更新：2026-07-09
> 状态：Phase 1 规划完成

---

## 需求结论

在已有创作区四层结构（总纲→卷纲→章纲→正文）手动 CRUD 基础上，补齐伏笔库持久化、正文库持久化、过审流程、提示词库，建立「程序驱动 AI」的完整数据基础。分三阶段推进：Phase 1 基础能力 → Phase 2 提示词库+AI接入 → Phase 3 AI全面赋能。

- 用户对话：2026-07-09 多轮需求讨论，核心理念「数据化管理小说创作」
- 当前约束：创作区四层结构手动 CRUD 已完成；伏笔库仅前端内存状态；正文库无持久化；AI 配置和基础调用链路已通

## 用户与场景

**目标用户**：网络小说作家（个人创作者）

**使用场景**：
- 长篇连载创作（10万-100万字级别）
- 需要管理大量角色、伏笔、设定的一致性
- 希望 AI 辅助生成/润色，但不希望 AI 理解错误题材或设定

**核心理念**：将小说创作从「人脑记忆」转为「数据管理」。程序管理所有数据的存储、提取、关联、流转；AI 被程序驱动，只在被调用时执行生成、提取、分析任务。

---

## 核心痛点

| 痛点 | 现状 | 解决方式 |
|------|------|----------|
| 角色/伏笔/事实靠人脑记忆 | 写到50章忘了第3章的伏笔 | 结构化数据落库，程序自动关联 |
| AI 不理解小说上下文 | 只给AI一段文字，它不知道角色是谁 | 程序从各库提取数据，填充提示词模板 |
| 大纲和正文割裂 | 写正文时无法对照章纲 | 章纲和正文在同一工作流中衔接 |
| 设定修改需要全文搜索替换 | 改了角色名字要手动找所有提及处 | 章纲只存角色ID，AI生成时实时取最新设定 |
| 前后文不一致难以发现 | 写了矛盾的情节自己不知道 | 过审时AI提取事实，可全局分析矛盾 |

## 系统架构概览

### 八大模块

| 模块 | 职责 | AI 参与 |
|------|------|---------|
| 书籍信息 | 题材/风格/受众等元数据 | 全程注入AI，防止误解题材 |
| 规则库 | 全局规则/设定规则/文笔规则 | 作为约束条件注入提示词 |
| 设定库 | 角色/物品/地点/势力/其他实体 | 注入正文生成；过审提取状态变更 |
| 创作区 | 总纲→卷纲→章纲→正文 | AI生成正文；过审提取数据 |
| 正文库 | 过审存稿（单向流） | AI润色/扩写/去味 |
| 伏笔库 | 埋/收/过期状态追踪 | 过审提取伏笔变更 |
| 事实一致性库 | 已发生情节事实记录 | 过审提取事实；全局矛盾分析 |
| 提示词库 | 按场景分类的变量模板 | 程序填充变量后发给AI |

### 关键设计原则

1. **程序驱动 AI**：所有数据提取、组装由程序完成，AI 只执行生成/提取/分析
2. **章纲只存ID**：设定库修改实时生效，不存快照
3. **提示词是变量模板**：程序从各库查询数据填入 `{{变量}}`，生成完整 prompt
4. **过审单次AI调用**：一次性提取事实+伏笔变更+角色状态变更，用户逐项确认
5. **正文库单向流**：过审后正文归档，正文库修改不回流章纲
6. **规则继承**：角色继承大类规则+自定义规则，不能冲突

## 功能范围

### Phase 1：基础能力（本期）

> 目标：用户可纯手动完成「总纲→卷纲→章纲→过审→正文归档」完整闭环

#### 1.1 伏笔库持久化
- 伏笔数据入库（SQLite），字段：id, bookId, name, description, status(hidden/revealed/expired), plantChapterId, revealChapterId, relatedCharacterIds, sortOrder
- 章纲中埋伏笔时自动写入伏笔库
- 章纲中回收伏笔时更新状态
- 伏笔库面板展示所有伏笔，按状态筛选

#### 1.2 正文库持久化
- 已有 `archived_chapters` 表，需确保完整对接
- 过审后正文副本自动归档
- 正文库面板展示已归档章节，可查看/编辑

#### 1.3 章纲过审流程（手动版）
- 章节状态流转：`planned → writing → done`
- 过审界面：用户手动确认/填写三项数据
  - 事实：手动添加本章发生的关键事实
  - 伏笔变更：手动标记新建伏笔/回收伏笔
  - 角色状态变更：手动标记角色属性/状态变化
- 确认后批量落库（事实库/伏笔库/设定库/正文库）
- 过审按钮仅在正文已写入时可用

#### 1.4 事实库
- 新建事实表：id, bookId, chapterId, chapterNumber, content, relatedCharacterIds, createdAt
- 过审时手动添加事实
- 事实库面板展示所有事实，按章节筛选
- 写新章纲时可查看前情事实列表

#### 1.5 卷纲内置工作台
- 卷纲编辑页下方显示统计：本卷章节数、关联角色数、伏笔数、事实数
- 点击可跳转到对应章节/角色/伏笔/事实详情
- 快速操作入口（如：从卷纲直接进入某一章的章纲编辑）

#### 1.6 规则引用（基础版）
- 设定库实体可引用规则库中的设定规则
- 引用关系存储：entity_rule_references 表（entityId, ruleId）
- 设定库编辑页显示已引用的规则

### Phase 2：提示词库 + AI 接入

> 目标：程序能从各库提取数据，填充模板，AI 可被驱动执行任务

#### 2.1 提示词库
- 场景分类（一级tab = 系统AI能力模块，二级tab = 具体场景）
- 模板存储：场景标识、模板内容（含 `{{变量}}` 占位符）、变量定义（名称、数据来源说明）
- 每个场景一份系统默认模板（不可删除），用户可复制创建自定义版本
- 变量预览：展示填充后的完整 prompt
- 模板测试：输入变量值预览结果，防止变量错误导致AI拿不到数据

#### 2.2 数据提取引擎
- 程序按场景从各库查询数据：
  - 书籍信息 → `{{genre}}`, `{{style}}` 等
  - 规则库 → `{{globalRules}}`, `{{writingRules}}`
  - 设定库 → `{{characterName}}`, `{{characterDesc}}` 等（按章纲关联的角色ID）
  - 事实库 → `{{previousFacts}}`（按章节号查询前情）
  - 伏笔库 → `{{foreshadowsToRecover}}`（状态=已埋的伏笔）
  - 章纲 → `{{chapterOutline}}`, `{{userOpening}}`, `{{userEnding}}`
- 填充模板变量，生成完整 prompt

#### 2.3 过审 AI 流程
- 程序组装 prompt（正文+章纲+角色列表+伏笔列表）
- AI 一次性返回结构化 JSON：`{facts, foreshadowChanges, characterStateChanges}`
- 用户在统一界面逐项确认/修改/拒绝
- 确认后程序批量落库

### Phase 3：AI 全面赋能

> 目标：创作区所有 AI 场景接通

- 总纲/卷纲 AI 辅助生成
- 正文 AI 生成（完整上下文组装）
- 正文库 AI 润色/扩写/去味
- 设定库 AI 规则检查（实体 vs 引用规则对比）
- 事实库全局矛盾分析

### 本期不做

- 回退系统（分支重写）
- 知识图谱
- 用户自定义工作流
- 卷纲/章纲 AI 数据分析图表

## 用户流程

### 完整创作流程

```
1. 创建书籍 → 填写书籍信息
2. 定义规则库（全局/设定/文笔规则）
3. 创建设定库实体（角色/物品/地点等），引用规则
4. 写总纲（开头/结尾/分卷规划）
5. 写第一卷卷纲（开头/结尾/阶段/内容）
6. 写章纲（关联角色/埋伏笔/场景/自定义首尾）
7. AI生成正文（Phase 2+）→ 用户确认
8. 过审 → 确认事实/伏笔/角色变更 → 批量落库
9. 写下一章...（重复6-8）
10. 一卷写完 → 写下一卷卷纲
```

### 过审流程（详细）

```
1. 用户在章纲编辑器中点击「过审」
2. 弹出过审确认界面，显示三项：
   a. 事实提取区：可添加/编辑/删除事实条目
   b. 伏笔变更区：可标记新建伏笔/回收伏笔
   c. 角色状态变更区：可标记角色属性变化
3. 用户逐项审核，可手动修改AI提取结果（Phase 2）或手动填写（Phase 1）
4. 点击「确认过审」
5. 程序批量落库：事实→事实库 / 伏笔→伏笔库 / 角色→设定库 / 正文→正文库
6. 章节状态 → done
7. 界面反馈：过审成功，可继续写下一章
```

## 状态与反馈

- **空状态**：各面板空状态引导（如伏笔库为空时提示"在章纲中标记伏笔"）
- **加载中**：各面板数据加载时显示骨架屏
- **操作成功**：保存/过审成功后轻提示（toast）
- **操作失败**：错误提示 + 原因说明
- **删除/危险操作**：二次确认弹窗（使用 `confirm-delete` 组件）
- **过审不可用**：正文未写入时过审按钮置灰 + tooltip提示

## 风险与待确认问题

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| AI提取的结构化JSON格式不稳定 | 过审流程中断 | 设计容错解析+用户手动修正 |
| 提示词变量过多导致组装复杂 | 数据提取引擎难维护 | 按场景拆分，每个场景独立组装函数 |
| 过审批量落库部分失败 | 数据不一致 | 事务处理，失败整体回滚 |
| 章纲关联数据过多影响性能 | 编辑卡顿 | 按需加载，统计数异步查询 |

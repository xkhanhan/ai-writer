# 第一阶段设计方案

## 1. 设计目标

第一阶段的设计目标是做一个“能跑通”的本地产品骨架，而不是提前建设完整写作系统。

本阶段只验证四件事：

- 页面能访问。
- 本地数据能保存。
- 后端接口能工作。
- AI 能被后端调用，结果能回到前端。

## 2. 信息架构

第一阶段只有一个主页面和两个弹窗。

```text
首页
  ├─ 书籍列表
  ├─ 创建新书弹窗
  ├─ AI 配置弹窗
  └─ AI 测试区域
```

## 3. 首页布局

建议首页分成三块：

```text
顶部栏
  ├─ 产品名称
  └─ AI 配置按钮

书籍区域
  ├─ 创建新书按钮
  ├─ 空状态
  └─ 书籍列表

AI 测试区域
  ├─ Prompt 输入框
  ├─ 测试按钮
  ├─ 返回结果
  └─ 错误提示
```

首页不要做成营销页。第一屏直接展示工具本体。

## 4. 交互流程

### 4.1 创建新书

```text
点击创建新书
  ↓
打开弹窗
  ↓
填写书名和简介
  ↓
前端校验
  ↓
POST /api/books
  ↓
成功后关闭弹窗
  ↓
刷新书籍列表
```

失败时：

- 弹窗不关闭。
- 表单区域展示错误。
- 用户可以修改后再次提交。

### 4.2 保存 AI 配置

```text
点击 AI 配置
  ↓
GET /api/ai/config
  ↓
打开配置弹窗
  ↓
填写或修改配置
  ↓
POST /api/ai/config
  ↓
显示保存成功或失败
```

注意：

- 获取配置时不显示完整 API Key。
- 如果已经配置过 Key，可以显示“已配置”状态。
- 用户重新输入 API Key 后再覆盖保存。

### 4.3 测试 AI

```text
输入测试 prompt
  ↓
点击测试
  ↓
POST /api/ai/chat
  ↓
按钮进入请求中状态
  ↓
展示 AI 返回内容或错误
```

测试 AI 前不强制要求已有书籍。

## 5. 前端状态设计

首页需要维护这些状态：

- 书籍加载中。
- 书籍加载失败。
- 书籍为空。
- 书籍列表。
- 创建新书提交中。
- 创建新书错误。
- AI 配置读取中。
- AI 配置保存中。
- AI 配置错误。
- AI 测试请求中。
- AI 测试结果。
- AI 测试错误。

## 6. 后端设计

后端分成三组接口：

- `books`：书籍列表和创建新书。
- `ai/config`：AI 配置读取和保存。
- `ai/chat`：AI 测试调用。

后端职责：

- 校验请求参数。
- 读写本地 JSON 数据。
- 隐藏完整 API Key。
- 调用 AI 平台。
- 返回统一错误格式。

## 7. 本地数据设计

第一阶段使用本地 JSON 文件，避免提前引入数据库。

建议目录：

```text
data/
  books.json
  ai-config.json
```

`books.json` 保存书籍列表。

`ai-config.json` 保存 AI 配置，不提交 Git。

后续进入第二阶段后，如果章节、正文和设定数据开始变多，再评估 SQLite 或 Prisma。

## 8. 错误处理设计

所有后端接口失败时返回：

```json
{
  "success": false,
  "error": "可读错误信息"
}
```

前端只展示 `error` 字段，不直接展示堆栈。

常见错误：

- 书名为空。
- 本地数据读取失败。
- 本地数据保存失败。
- AI 配置缺失。
- AI 平台请求失败。
- AI 返回空内容。

## 9. 安全边界

第一阶段必须守住：

- API Key 不进入前端状态。
- API Key 不返回给前端。
- API Key 不提交 Git。
- 前端只请求本地后端。
- 后端统一代理 AI 请求。

## 10. 实现顺序建议

第一阶段实现顺序：

1. 建立基础工程。
2. 建立本地数据读写工具。
3. 实现书籍列表接口。
4. 实现创建新书接口。
5. 实现 AI 配置接口。
6. 实现 AI 测试接口。
7. 实现首页 UI。
8. 串联前后端。
9. 手动验收错误状态。

## 11. 本阶段不做的设计

本阶段不设计：

- 创作编辑器布局。
- 章节树。
- 提示词模板。
- 工作流编排。
- AI 上下文包。
- 状态事件。
- 验收池。
- 回滚机制。

# AI 系统架构设计

> 最后更新：2026-07-10
> 状态：设计文档，指导后续开发

---

## 一、核心理念

**将小说创作从"人的记忆"变成"系统的数据"。**

- 所有书籍信息（角色、规则、伏笔、事实）通过程序管理
- AI 基于完整数据工作，而非残缺上下文
- 数据和提示词解耦：开发者写提示词模板，程序自动组装数据

## 二、数据模块总览

```
抽象层（定义世界基础约束）
├── 书籍信息：题材、风格、目标受众 → 每次 AI 调用注入
├── 规则库：
│   ├── 全局规则：世界观铁律（政治合规等）→ 每次注入
│   ├── 写作规则：文笔要求、禁忌表达 → 每次注入
│   └── 设定规则：绑定到实体类别（角色规则→角色大类，物品规则→物品大类）
│
实体层（定义世界具体内容）
├── 设定库：角色/物品/地点/势力/其他 → 章纲引用，AI 生成时注入
├── 伏笔库：埋入/收回/状态跟踪 → 过审时更新
├── 事实库：AI 提取的已发生事件 → 过审时提取，一致性检查用
│
执行层（实际写作流程）
├── 创作区（三阶段）
│   ├── 总纲：书的骨架（开头、结尾、几个卷）
│   ├── 卷纲：卷的骨架（阶段、涉及角色/伏笔/事实统计）
│   └── 章纲：章的蓝图（场景、人物、事件、伏笔、自定义开头结尾）
├── 正文库：过审后的存档（只读，单向数据流）
└── AI 能力：生成/润色/去AI味/检查/提取
```

## 三、数据流图

```
书籍信息 ←→ 规则库
     ↓
  创作区（总纲 → 卷纲 → 章纲）
     ↓
  AI 生成正文（注入：章纲+角色+规则+事实+伏笔+前文）
     ↓
  过审流水线（一次性提取 → 用户确认 → 数据落库）
     ├──→ 事实库（提取事实）
     ├──→ 伏笔库（埋入/收回伏笔）
     ├──→ 设定库（更新角色/物品状态）
     └──→ 正文库（存档正文）
     ↓
  AI 润色/扩写/去AI味（仅正文+规则，不重新注入全部上下文）
     ↓
  最终稿
```

### 单向数据流原则

- **创作区 → 正文库**：过审后正文复制到正文库（单向）
- **正文库修改不影响章纲**：正文库的润色/修改不反向影响创作区
- **创作区修改重新生成**：修改章纲后重新生成正文，过审后更新正文库

## 四、AI 能力清单

### 4.1 生成类

| 能力 | 调用时机 | 输入 | 输出 |
|------|---------|------|------|
| 正文生成 | 章纲完成后 | 章纲+角色+规则+事实+伏笔+前文 | 正文 |
| 书籍简介扩写 | 书籍信息页 | 书籍简介 | 扩写后简介 |
| 章纲生成 | 卷纲完成后 | 卷纲+已有章纲+角色 | 章纲建议 |

### 4.2 处理类

| 能力 | 调用时机 | 输入 | 输出 |
|------|---------|------|------|
| 润色 | 正文库 | 正文+全局规则+写作规则 | 润色后正文 |
| 去 AI 味 | 正文库 | 正文 | 去 AI 味后正文 |
| 扩写 | 正文库 | 正文+章纲摘要 | 扩写后正文 |

### 4.3 检查类

| 能力 | 调用时机 | 输入 | 输出 |
|------|---------|------|------|
| AI 检查角色 | 任何时候 | 角色设定+规则+事实 | 问题列表 |
| 事实一致性检查 | 过审后 | 所有事实+规则+设定 | 矛盾列表 |
| 伏笔状态检查 | 过审后 | 所有伏笔+章纲 | 未收回伏笔列表 |

### 4.4 提取类（过审专用）

| 能力 | 调用时机 | 输入 | 输出 |
|------|---------|------|------|
| 事实提取 | 过审 | 正文+章纲 | 结构化事实列表 |
| 伏笔变更提取 | 过审 | 正文+现有伏笔 | 伏笔埋入/收回列表 |
| 角色状态提取 | 过审 | 正文+角色设定 | 角色状态变更列表 |
| 物品状态提取 | 过审 | 正文+物品设定 | 物品状态变更列表 |

## 五、过审流水线（一次性提取）

**关键设计：AI 一次性提取所有数据，用户统一批量确认，非步骤型流程。**

```
用户点击"过审"
  ↓
ContextBuilder 组装过审上下文
  ├── 正文内容
  ├── 章纲信息
  ├── 现有角色设定
  ├── 现有伏笔
  └── 世界规则
  ↓
AI 一次性返回结构化 JSON
  {
    facts: [...],           // 提取的事实
    foreshadowChanges: [...], // 伏笔变更（埋入/收回）
    characterStates: [...],  // 角色状态变更
    itemStates: [...],       // 物品状态变更
  }
  ↓
前端渲染结构化结果
  ┌─────────────────────────────────────┐
  │  AI 过审结果            [全选] [取消]│
  ├─────────────────────────────────────┤
  │  📋 提取的事实 (5条)                │
  │  ☑ 张三进入山洞发现符文             │
  │  ☑ 李四怀疑张三修炼邪功             │
  │  ☐ 王五暗中观察（不确定）            │
  ├─────────────────────────────────────┤
  │  📌 伏笔变更 (2条)                  │
  │  ☑ 埋入: 山洞符文与铁剑有关联        │
  │  ☑ 收回: 张三夜探后山（第3章埋入）    │
  ├─────────────────────────────────────┤
  │  👤 角色状态 (3条)                   │
  │  ☑ 张三: 位置→山洞, 新知→符文存在    │
  │  ☑ 李四: 关系→怀疑升级              │
  │  ☐ 王五: 新增监视行为               │
  ├─────────────────────────────────────┤
  │  [确认采纳选中项]                    │
  └─────────────────────────────────────┘
  ↓
用户勾选确认 → 数据落库
  ├── 事实 → 事实库
  ├── 伏笔变更 → 伏笔库
  ├── 角色状态 → 设定库
  ├── 物品状态 → 设定库
  └── 正文 → 正文库（复制存档）
```

### 过审的 token 预算

```
总预算: ~8000 tokens
├── System Prompt:     500 tokens (提取指令+JSON格式要求)
├── 正文内容:         3000 tokens (核心数据)
├── 章纲信息:         1000 tokens (任务上下文)
├── 现有角色设定:     1500 tokens (对比参照)
├── 现有伏笔:         500 tokens (对比参照)
├── 世界规则:         500 tokens (一致性约束)
└── 预留输出:        1000 tokens (结构化JSON)
```

## 六、ContextBuilder 设计

### 6.1 多阶段组装流水线

```
Stage 1: 数据收集（从 DB 加载所有相关信息）
Stage 2: 数据筛选（按 token 预算裁剪）
Stage 3: 知识库注入（正反例、写作规范）
Stage 4: 模板渲染（$变量替换）
Stage 5: 最终校验（token 总量检查）
```

### 6.2 各 AI 能力的 ContextBuilder

```typescript
// 正文生成
buildContentGenerationContext(bookId, chapterId) → {
  systemPrompt,     // 写作规范 + 世界规则（硬约束）
  userPrompt,       // 章纲+角色+事实+伏笔+前文
  tokenBudget: 8000
}

// 过审提取
buildReviewContext(bookId, chapterId) → {
  systemPrompt,     // 提取指令 + JSON格式要求
  userPrompt,       // 正文+章纲+现有设定
  tokenBudget: 8000
}

// AI 检查角色
buildCharacterAuditContext(bookId, characterId) → {
  systemPrompt,     // 检查指令
  userPrompt,       // 角色设定+规则+事实
  tokenBudget: 4000
}

// 润色/去AI味（简单，不需要大量上下文）
buildTextProcessingContext(bookId, text, functionKey) → {
  systemPrompt,     // 处理指令
  userPrompt,       // 正文+全局规则
  tokenBudget: 4000
}
```

### 6.3 角色状态自动推导

```typescript
// 不是手动维护，而是从事实自动推导
deriveCharacterState(characterId, upToChapter) → {

}
```

## 七、提示词系统

### 7.1 提示词库结构

```
提示词库
├── 按场景分类（一级 Tab）
│   ├── 书籍信息
│   │   └── 书籍简介扩写（二级 Tab）
│   ├── 创作区
│   │   ├── 正文生成
│   │   ├── 润色
│   │   ├── 去 AI 味
│   │   └── 事实提取
│   └── 正文库
│       ├── 润色
│       └── 扩写
└── 每个提示词包含：
    ├── 基础模板（不可删除，可复制）
    ├── 用户自定义版本（可激活/停用）
    ├── 变量列表（$变量名 → 数据来源说明）
    ├── 变量检查（验证替换是否正确）
    └── 测试能力（发送测试数据，查看输出）
```

### 7.2 变量系统

提示词模板中的 `$variable` 由程序自动替换为实际数据：

```
模板: "书名：$bookTitle，题材：$bookGenre"
数据: { bookTitle: "青云诀", bookGenre: "玄幻" }
结果: "书名：青云诀，题材：玄幻"
```

**变量来源必须透明** — 用户看到 `$chapterSummary`，必须知道数据来源是"章纲的摘要字段"。

**变量替换失败处理** — 如果变量无数据，跳过该段落 + 提示用户。

### 7.3 提示词设计原则

#### System Prompt = 固定规范 + 世界规则

```markdown
# 角色
你是一位专注于玄幻/仙侠题材的网络小说作家...

# 写作红线（硬约束）
1. 世界规则中标记的规则不可违反
2. 人物性格必须与设定一致
3. 不使用现代网络用语
4. 不出现 AI 常见表达
5. 章纲中的每个场景必须出现
6. 伏笔必须按要求埋入或呼应

# 写作指南（参考）
- 场景描写：五感细节
- 对话：符合人物身份
- 节奏：紧张-舒缓交替
```

#### User Prompt = 海量上下文（按层级组织）

```markdown
# 章纲信息（核心任务目标）
标题、摘要、场景列表、出场人物、目标字数

# 角色档案（写这个场景涉及的人物）
姓名、性格、当前状态、已知信息、语言风格

# 事实库（已发生的事件，不可矛盾）
按时间线 + 人物双维度索引

# 世界规则（必须遵守）
全局规则 + 写作规则

# 伏笔（需要呼应或埋入）
当前活跃伏笔列表

# 前文衔接
上一章末尾内容
```

#### Token 预算分配

```
总预算: 8000 tokens
├── System Prompt:     500 tokens  (6%)
├── 章纲信息:         1000 tokens (12%)
├── 角色档案:         2000 tokens (25%)
├── 事实库:           2000 tokens (25%)
├── 世界规则:          500 tokens (6%)
├── 伏笔:             300 tokens (4%)
├── 前文衔接:          500 tokens (6%)
└── 预留输出:         1200 tokens (15%)
```

超预算时按优先级截断：章纲/规则不可截断 → 事实库取最近相关 → 角色档案精简次要角色

## 八、回退机制（Phase 2 规划）

### Phase 1: 标记问题 + 强制重新过审

修改第 N 章后，第 N+1 章及之后的章节标记为"需要重新过审"，用户逐章检查。

### Phase 2: 分支回退（远期）

类似 git，创建回退分支，主线从第 N 章重写，旧分支保留可查看。

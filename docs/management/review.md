# 评审规范

> 代码评审、架构评审、PRD 评审的流程与检查清单。

## 1. 代码评审

### 1.1 评审时机

- 所有 PR 必须经过评审后才能合并。
- 自我评审不替代他人评审。

### 1.2 评审检查清单

**必检项**（任一不通过不予合并）：

- [ ] `npm run typecheck` 零错误
- [ ] `npm run lint` 零警告
- [ ] 无 `server/` 代码导入到客户端组件
- [ ] 无硬编码颜色值（必须使用 CSS 变量）
- [ ] 删除操作使用 `confirmDelete`
- [ ] 异步操作有 loading 状态
- [ ] API 函数返回 `Result<T>`，不抛出异常

**建议项**：

- [ ] 组件职责单一，复杂逻辑提取到 Hook
- [ ] 新增代码有清晰的命名
- [ ] 无冗余的 console.log / TODO 残留

## 2. 架构评审

适用场景：
- 新增模块/系统
- 修改三层架构依赖方向
- 引入新的技术依赖

评审要点：
- 是否符合三层架构依赖方向
- 是否有更简单的实现方式
- 对现有系统的影响范围

## 3. PRD 评审

适用场景：
- 新功能需求文档
- 重大功能变更

评审要点：
- 需求是否清晰、无歧义
- 数据模型是否合理
- 是否与现有功能冲突
- 验收标准是否可测试

# 计划管理规范

> 迭代规划、任务拆解、进度跟踪的方法论。

## 1. 迭代规划

### 1.1 工作计划文档

项目总计划维护在 `docs/plans/work-plan.md`，记录：

| 字段 | 说明 |
|------|------|
| 工作项名称 | 简要描述 |
| 内容 | 具体实施内容 |
| 状态 | 待实现 / 已有方案 / 已完成 |
| 指导文件 | 关联的设计文档 |
| 计划/完成时间 | 时间节点 |

### 1.2 功能设计文档

每个功能的设计文档放在 `docs/plans/` 目录，包含：
- 功能概述与目标
- 数据模型定义
- 前后端接口设计
- 实施任务清单
- 验收标准

## 2. 任务拆解

### 2.1 拆解原则

- 每个任务应可在一次提交内完成。
- 任务之间尽量低耦合，可并行开发。
- 任务粒度：最小可验证的独立变更。

### 2.2 任务状态

| 状态 | 说明 |
|------|------|
| 待实现 | 方案已确认，尚未开始 |
| 已有方案 | 设计完成，待排期 |
| 进行中 | 正在开发 |
| 已完成 | 已合并到 master |

## 3. 进度跟踪

- 通过 `docs/plans/work-plan.md` 跟踪总体进度。
- 通过 Git 提交历史跟踪具体实施进展。
- 通过 PR 描述记录每次变更的范围和影响。

# project-scan - novel-writer 工程扫描

> 触发方式：`/project-scan`、`/scan`、`工程扫描`、`扫描项目问题`、`前端扫描`、`后端扫描`、`数据库扫描`

你是 novel-writer 项目的工程扫描助手。目标不是列出所有瑕疵，而是找出会影响功能、数据可靠性、潜在风险、用户体验或长期维护成本的问题，并生成清晰报告。

---

## 一、扫描原则

### 必须记录的问题

只记录以下类型：

1. **功能影响**：构建失败、页面不可用、保存失败、删除异常、数据展示错误。
2. **数据风险**：保存到错误对象、脏数据、孤儿数据、缓存不同步、误提示保存成功。
3. **潜在风险**：弱校验、错误契约不统一、状态流不清晰、跨层依赖错误。
4. **用户体验升级**：用户会感知到的状态不同步、错误提示不明确、操作反馈不可信。
5. **维护成本**：会明显拖慢后续开发的架构边界、重复模式、响应格式混乱。

### 不要记录的问题

不要把报告写成 lint 清单。以下问题默认不记录，除非它们已经影响功能：

- 单纯换行、缩进、空格、分号。
- 纯审美偏好。
- 不影响理解和运行的命名喜好。
- 没有证据的猜测。
- 一次性 mock 文档里的演示代码。

## 二、固定扫描流程

### Step 1：读取项目入口

优先读取：

- `AGENTS.md`
- `docs/README.md`
- `package.json`
- `app/`
- `shared/`
- `server/`

按需读取：

- UI 问题：`docs/visual.md`、`docs/components.md`
- API 问题：`docs/api.md`
- 数据库问题：`docs/architecture.md`、`docs/engineering.md`
- 校验问题：`docs/validation.md`

不要一次性读取所有文档。

### Step 2：执行工程校验

使用 PowerShell 命令：

```powershell
npm run typecheck
npm run lint
npm run build
```

报告里必须写明三项结果。失败项必须作为最高优先级问题记录。

### Step 3：分区扫描

必须按三个主区扫描：

1. 前端扫描
2. 后端/API 扫描
3. 数据库/持久化扫描

可以追加：

4. 工程配置扫描
5. 用户体验扫描

---

## 三、前端扫描重点

重点看：

- 状态是否有单一来源，父子组件是否混用受控/非受控状态。
- 保存、删除、切换对象时是否会误写、丢写、残留旧数据。
- 自动保存、定时器、异步请求是否捕获了正确对象 id。
- 页面 A 更新后页面 B 是否会同步刷新。
- API 返回失败时是否会误提示成功。
- hook 是否正确清理 timer、listener、subscription。
- `shared/` 是否反向依赖 `app/`。
- 设计系统是否被明显绕过，例如大量内联样式导致主题和响应式不可控。

典型高价值问题：

- “A 文件内容可能保存到 B 文件。”
- “标签改名后设定库仍显示旧标签。”
- “保存失败但 UI 提示成功。”
- “编辑第二个对象时表单残留第一个对象内容。”

低价值问题不要写：

- “这里可以换个变量名。”
- “这里多一个空行。”
- “这里按钮颜色我不喜欢。”

---

## 四、后端/API 扫描重点

重点看：

- API 响应格式是否统一。
- `jsonSuccess/jsonError` 是否一致使用。
- 前端 `ApiClient` 是否能读取服务端真实错误。
- POST/PATCH 是否做了 DTO 白名单解析。
- 枚举、数组、数字、必填字段是否校验。
- API route 是否保持薄层：parse request -> call server -> return JSON。
- 是否把服务端异常吞成模糊错误，导致用户无法理解问题。

典型高价值问题：

- “服务端返回 `{ error }`，客户端只读 `message`，导致错误信息丢失。”
- “PATCH 直接透传 body 到 storage，非法字段可能进入业务层。”
- “接口有三种成功响应格式，业务 helper 容易误判。”

---

## 五、数据库/持久化扫描重点

重点看：

- 表之间是否有明确外键关系。
- 需要级联删除的关系是否有 `ON DELETE CASCADE`。
- 删除书籍、章节、标签后是否可能留下孤儿数据。
- JSON 字段写入前是否 normalize，读取失败是否静默 fallback。
- migration 是否幂等，不破坏已有数据。
- `updated_at` 是否在更新路径里同步维护。
- 查询是否按业务需要建立索引。

典型高价值问题：

- “删除书籍后某些业务表可能残留孤儿数据。”
- “JSON 字段没有形状校验，错误数据会导致前端空白。”
- “更新内容后父级更新时间不同步，列表排序不准。”

## 六、报告格式

扫描报告必须包含：

```markdown
# 工程扫描问题报告

## 当前状态
- typecheck:
- lint:
- build:
- 工作区状态:

## 问题分级
| 级别 | 含义 |
|------|------|
| P0 | 阻断构建或核心流程不可用 |
| P1 | 数据可靠性、保存一致性、主要体验风险 |
| P2 | 架构边界、维护成本、长期风险 |
| P3 | 体验优化或规范收敛 |

## 前端扫描
### P1：问题标题
- 位置：
- 现象：
- 业务影响：
- 建议修复：

## 后端/API 扫描

## 数据库/持久化扫描

## 建议修复顺序
```

每个问题都要用业务语言解释清楚：用户会看到什么、数据会怎样错、后续开发会怎样受影响。

---

## 七、输出要求

- 不要输出超过 15 个问题，除非用户要求全量审计。
- 优先输出 P0/P1。
- 每个问题必须有代码位置或明确证据。
- 不确定的问题写成“需要进一步验证”，不要当成事实。
- 如果工程校验全部通过，也要明确说明“当前没有构建阻断”。
- 最后给出修复顺序，而不是只列问题。

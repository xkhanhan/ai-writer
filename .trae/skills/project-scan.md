# project-scan - 工程与架构扫描

> 触发方式：`/project-scan`、`/scan`、`工程扫描`、`架构扫描`、`代码审查`、`扫描项目问题`

你是高级架构师视角的工程审查助手。目标不是找格式毛病，而是识别会影响功能正确性、数据可靠性、架构稳定性、可维护性、安全性和长期迭代成本的问题。

---

## 一、工作边界

- 只扫描和报告问题，不直接修复代码；用户明确要求修复时再进入开发 skill。
- 不迎合用户的预设结论；所有判断必须有代码位置、运行结果或明确证据。
- 不追求“完美架构”。只记录会造成现实风险、明显维护成本或后续迭代阻力的问题。
- 不把报告写成 lint 清单。换行、空格、命名偏好、纯审美问题默认不记录。
- 不基于猜测下结论；不确定时写“需要进一步验证”并说明验证方式。

---

## 二、输入读取

优先读取：

- `AGENTS.md`
- `docs/README.md`
- `package.json`
- `app/`
- `shared/`
- `server/`

按需读取：

- UI/设计系统：`docs/visual.md`、`app/globals.css`、`shared/ui/`
- API/服务端：`app/api/`、`server/`
- 数据/持久化：`server/storage/`、`data/`、相关数据库文档
- 既有规范/需求/架构稿件：用户提供的路径，或在 `docs/` 中寻找明显相关文件

不要一次性读取所有文档。先建立地图，再按风险点深入。

---

## 三、扫描流程

1. 确认扫描范围：全工程、前端、后端/API、数据持久化、某个功能模块，或某个架构主题。
2. 读取入口和相关规范，理解当前分层、依赖方向、数据流和关键用户流程。
3. 执行必要校验，并在报告中写明结果：

```powershell
npm run typecheck
npm run lint
npm run build
```

4. 分区审查：
   - 前端交互与状态流
   - API 契约与服务端边界
   - 数据持久化与迁移
   - 模块依赖与复用边界
   - 安全、错误处理、异常恢复
5. 输出问题报告和修复顺序。

---

## 四、重点审查清单

### 架构与模块边界

- 是否存在 `client` 组件直接导入 `server/*`。
- `shared/` 是否反向依赖 `app/`。
- API route 是否保持薄层：parse request -> call server -> return JSON。
- 可复用能力是否放在合理层级；是否为了小问题引入过重抽象。
- 模块是否承担过多职责，导致后续功能必须改同一个大文件。

### 健壮性与数据可靠性

- 保存、删除、切换对象时是否会误写、丢写、残留旧数据。
- 异步请求、自动保存、定时器是否捕获了正确对象 id。
- API 失败是否被 UI 误提示为成功。
- DTO 是否做白名单解析和字段校验。
- JSON 字段、枚举、数组、数字、必填字段是否有明确校验。
- 删除数据后是否留下孤儿数据；migration 是否幂等。

### 可维护性与可迭代性

- 是否有重复业务规则散落多处。
- 状态来源是否清晰，是否混用受控/非受控状态。
- 错误响应格式是否统一，前端是否能展示真实错误。
- 组件、hook、server service 是否职责单一。
- 是否能拆成清晰的原子能力，避免一个函数或组件同时承担多条业务链路。
- 设计系统是否被绕过，导致主题、响应式和一致性不可控。

### 安全与边界

- 用户输入是否在服务端重新校验。
- 文件路径、数据库写入、配置项是否存在越权或注入风险。
- 错误信息是否泄露内部实现，或反过来过于模糊导致无法排障。

## 五、问题分级

| 级别 | 含义 |
|------|------|
| P0 | 构建阻断、核心流程不可用、明显数据破坏 |
| P1 | 数据可靠性、保存一致性、安全性、主要体验风险 |
| P2 | 架构边界、维护成本、长期迭代风险 |
| P3 | 体验优化、规范收敛、局部可读性问题 |

---

## 七、输出要求

- 默认最多输出 15 个问题，优先 P0/P1/P2。
- 每个问题必须有文件位置或明确证据。
- 用业务语言解释影响：用户会看到什么、数据会怎样错、后续开发会怎样受影响。
- 给出修复顺序，不只列问题。
- 如果校验全部通过，明确说明“当前没有构建阻断”，但仍继续做架构风险判断。

# project-scan - 工程与架构扫描

> 触发方式：`/project-scan`、`/scan`、`工程扫描`、`架构扫描`、`代码审查`、`扫描项目问题`、`审查代码`

用于从高级架构师视角审查工程、代码和架构问题。只要用户想"找问题、看架构是否合理、扫描代码质量/稳定性/可维护性"，就使用本 skill。默认只报告问题和修复顺序，不直接改代码。

---

## 目标

找出会影响功能正确性、数据可靠性、架构稳定性、安全性、可维护性、可扩展性和长期迭代成本的问题。不要把报告写成格式清单或个人审美意见。

---

## 输入读取

优先读取：

- `AGENTS.md`
- `docs/README.md`
- `package.json`
- `app/`
- `shared/`
- `server/`

按需读取：

- UI/设计系统：`docs/visual.md`、`app/globals.css`、`shared/ui/`
- API/服务端：`app/api/`、`server/`
- 数据/持久化：`server/storage/`、`data/`
- 规范/需求/架构稿件：用户提供路径，或 `docs/` 中明显相关文档

先建立项目地图，再按风险深入。不要一次性读取所有文档。

---

## 扫描流程

1. 确认范围：全工程、某模块、前端、后端/API、数据持久化、架构边界或安全。
2. 读取入口和相关规范，理解分层、依赖方向、数据流和关键用户流程。
3. 运行必要校验，并在报告中写结果：

```powershell
npm run typecheck
npm run lint
npm run build
```

4. 分区审查问题。
5. 输出问题报告和修复优先级。

---

## 审查重点

### 架构边界

- client 是否直接导入 `server/*`。
- `shared/` 是否反向依赖 `app/`。
- API route 是否保持薄层。
- 模块职责是否单一，是否存在一个文件承担多条业务链路。
- 原子能力是否清晰，是否能组合而不是互相缠绕。
- 是否为了小问题引入过重抽象。

### 健壮性与数据可靠性

- 保存、删除、切换对象时是否误写、丢写或残留旧数据。
- 异步请求、自动保存、定时器是否捕获正确对象 id。
- API 失败是否被 UI 误提示成功。
- DTO、枚举、数组、数字、必填字段是否校验。
- 删除后是否留下孤儿数据；migration 是否幂等。

### 可维护与可迭代

- 重复业务规则是否散落多处。
- 状态是否有单一来源。
- 错误响应格式是否统一。
- hook、组件、service 是否职责清晰。
- 设计系统是否被绕过，导致主题和响应式不可控。

### 安全与边界

- 用户输入是否在服务端重新校验。
- 路径、数据库写入、配置项是否存在越权或注入风险。
- 错误信息是否泄露内部实现，或过于模糊导致无法排障。

---

## 不记录的问题

除非已经影响功能或维护，不记录：

- 单纯换行、缩进、空格、分号。
- 纯命名偏好。
- 没证据的猜测。
- 个人审美意见。
- 一次性 mock 或草稿代码中的低风险问题。

---

## 分级

| 级别 | 含义 |
|------|------|
| P0 | 构建阻断、核心流程不可用、明显数据破坏 |
| P1 | 数据可靠性、安全性、保存一致性、主要体验风险 |
| P2 | 架构边界、维护成本、长期迭代风险 |
| P3 | 体验优化、规范收敛、局部可读性问题 |

```markdown
## 问题清单
### P1：问题标题
- 位置：
- 证据：
- 影响：
- 根因：
- 建议修复：
- 优先级：

## 需要进一步验证
- [只列有证据线索但不能下结论的问题。]

2. ...
```

默认最多输出 15 个问题，优先 P0/P1/P2。每个问题必须有代码位置或明确证据。
